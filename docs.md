# Documentation

## Архитектура

Проект разделён на два слоя:

- Backend (FastAPI)
- Frontend (Vue 3)

---

## Backend

Основные компоненты:

- `excel_service`  
  Чтение XLSX и валидация структуры

- `extraction_service`  
  Извлечение:
  - бренд
  - модель
  - цвет
  - гарантия

- `vendor_service`  
  Получение эталонных моделей через API:

/vendor/watch-models


- `matching_service`  
Сопоставление моделей:
- нормализация строк
- очистка токенов от шума
- построение сигнатуры модели (brand, family, generation, variant)
- сопоставление по сигнатурам (signature-based matching)
- ограниченный fallback для слабых моделей

---

## Frontend

Компоненты:

- `FileUpload.vue`  
Загрузка файла или ввод URL

- `ResultsTable.vue`  
Отображение результата

- `api.js`  
Работа с backend API

---

## Поток данных

1. Пользователь загружает файл или вводит URL  
2. Frontend отправляет данные на backend  
3. Backend:
 - читает Excel
 - извлекает данные
 - получает эталон
 - сопоставляет модели  
4. Результат возвращается на frontend  
5. Таблица отображается пользователю

## Подход к сопоставлению

В проекте используется сигнатурный подход вместо fuzzy matching.

Каждая модель преобразуется в структуру:

(brand, family, generation, variant)

Примеры:
- Samsung Watch 6 Classic → (samsung, watch, 6, classic)
- Apple Watch Series 11 → (apple, series, 11)
- Huawei Watch Fit 4 Pro → (huawei, fit, 4, pro)

### Логика:
1. точное совпадение нормализованных строк
2. сопоставление по сигнатурам
3. fallback только для слабых моделей

### Принципы:
- не сопоставляем Unknown бренд
- не сопоставляем аксессуары
- не смешиваем поколения
- не схлопываем версии (Pro ≠ base)

Приоритет — точность, а не количество совпадений.
